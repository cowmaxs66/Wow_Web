# Agent 执行规范

## 核心原则
- 先规划，后实施。
- 能小改，绝不大改。
- 能兼容，绝不推翻。
- 不写垃圾代码，不破坏既有成果，不输出无价值内容。
- 每一步必须可编译、可验证、可记录、可交接。

## 文档同步
每个推进步骤至少考虑更新：
- `计划报告/步骤执行记录.md`
- `计划报告/项目进度.md`
- `计划报告/项目总阶段预览表.md`
- `计划报告/变更记录.md`
- `计划报告/风险与问题记录.md`

## 状态枚举
只能使用以下状态：
- 未开始
- 进行中
- 已完成
- 已阻塞
- 已废弃
- 已归档

## 代码要求
- 优先最简单可行方案。
- 函数职责单一，避免过度抽象。
- 核心逻辑必须有中文注释，注释说明原因、输入、输出和边界。
- 禁止无用代码、占位代码、未使用变量、无实现 TODO。
- 代码必须按目录和模块职责归类，禁止把无关职责塞进同一文件。
- 单文件代码过长时优先按职责拆分，入口文件只做流程串接。
- 每个实现模块的 README 必须说明当前目录排版和文件职责。

## 执行循环
```text
完成一步 -> 编译检查 -> 调试/烟测 -> 更新双 MD -> 清理残留 -> 进入下一步
```

## GitHub 备份策略
- 本地先形成可验证提交。
- 远端仓库未确认前，不强行创建公开仓库。
- 默认建议使用 private 仓库保存开发进度。
