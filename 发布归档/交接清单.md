## 必须保留的工程约束
- 代码按模块拆分，禁止把多个职责塞进单一大文件。
- 核心逻辑必须有中文注释，注释要说明目的、输入输出和边界。
- 每个阶段最多两步必须更新一次文档。
- 每个可验证阶段必须提交并推送到 GitHub。
- 公开仓库禁止提交密钥、真实账号、商业脚本、私有资源和大漠二进制。

## 继续开发前检查
1. 确认 `git status --short` 干净。
2. 阅读 `计划报告/项目进度.md` 和 `计划报告/风险与问题记录.md`。
3. 阅读即将修改模块的 README 与技术设计文档。
4. 先确定扩展属于哪个模块，再决定是否新增接口或数据结构。
5. 修改后执行对应构建、测试和烟测。

## 交接结论
v1.0.0 是第一轮源码与文档可交接版本。它验证了 Client、Lua、DmBridge、Server、Web Admin 和脚本安全门的最小闭环，但尚不是生产部署版本。
