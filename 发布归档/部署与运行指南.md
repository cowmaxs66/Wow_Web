# 部署与运行指南

## 环境要求
| 类别 | 要求 |
|------|------|
| 操作系统 | Windows |
| Rust | workspace 当前声明 `rust-version = 1.92` |
| Node.js | 能运行 `npm install` 和 `npm run build` |
| Delphi | DmBridge 编译需要 `C:\RAD13\bin\dcc32.exe` |
| 大漠插件 | v1.24.0 起总包和 Client 分包携带 `dm.dll`、`RegDll.dll`；授权资料仍由本机自行准备，不进入仓库 |

## 初始化
```powershell
git clone https://github.com/cowmaxs66/Wow_Web.git
cd Wow_Web
cargo test --workspace
cd 实现模块/web-admin
npm install
npm run build
```

## 启动 Management Server
```powershell
cargo run -p management-server
```

发布包中可直接运行：

```powershell
.\management-server.exe
```

默认监听：

```text
127.0.0.1:18080
```

可通过环境变量覆盖：

```powershell
$env:MANAGEMENT_SERVER_BIND='127.0.0.1:18085'
cargo run -p management-server
```

启用本地 JSONL 历史持久化：

```powershell
$env:MANAGEMENT_SERVER_HISTORY_PATH='data/status-history.jsonl'
cargo run -p management-server
```

启用本地 JSONL 审计持久化：

```powershell
$env:MANAGEMENT_SERVER_AUDIT_PATH='data/server-audit.jsonl'
cargo run -p management-server
```

说明：
- v1.9.0 起，根目录 `management-server.exe` 是无控制台 GUI launcher，会隐藏启动 `bin\management-server-core.exe` 并打开 Web 管理页。
- 自动化或服务化场景使用 `.\bin\management-server-core.exe --no-open-browser`。
- v1.12.0 起，Client 状态上报会在 Server core 控制台打印 `Client 上线`、`Client 状态刷新` 或 `Client 离线`。如果需要看这些日志，必须运行 `bin\management-server-core.exe --no-open-browser`，不要运行无控制台 GUI launcher。
- 不设置 `MANAGEMENT_SERVER_HISTORY_PATH` 时，Server 保持纯内存历史。
- 设置后，状态上报会追加到 JSONL 文件。
- Server 重启时会从该文件恢复最新状态和每个 Client 最近 50 条历史。
- 不设置 `MANAGEMENT_SERVER_AUDIT_PATH` 时，Server 只保留进程内最近审计事件。
- 设置后，Server 消息、Client 命令和命令回执会追加到审计 JSONL，重启后可回放最近事件。
- JSONL 文件可能包含 Client 运行信息、命令类型和执行摘要，不得提交到 GitHub。

挂载 Web Admin 静态目录：

```powershell
$env:MANAGEMENT_SERVER_WEB_DIR='实现模块/web-admin/dist'
cargo run -p management-server
```

浏览器访问：

```text
http://127.0.0.1:18080
```

v1.5.0 起，正式 release 构建会把 Web Admin 内嵌到 `management-server.exe`。不配置 `MANAGEMENT_SERVER_WEB_DIR` 时，Server 会优先使用 exe 内嵌 Web；配置该变量时，外部目录优先。

## 启动 Client Agent
源码维护模式可执行一次 Lua bootstrap 并输出状态：

```powershell
cargo run -p client-agent -- --run-once
```

发布包中可直接运行：

```powershell
.\client-agent.exe
```

v1.9.0 起，根目录 `client-agent.exe` 是无控制台 GUI launcher，会隐藏启动 `bin\client-agent-core.exe --tray`，再由托盘拉起 monitor。

启用 Server 上报：

```powershell
$env:CLIENT_AGENT_SERVER_ENABLED='1'
$env:CLIENT_AGENT_SERVER_HOST='127.0.0.1'
$env:CLIENT_AGENT_SERVER_PORT='18080'
cargo run -p client-agent -- --run-once
```

常驻监控模式：

```powershell
.\bin\client-agent-core.exe --monitor
```

常用本机入口：

```powershell
.\bin\client-agent-core.exe --setup
.\bin\client-agent-core.exe --open-log
.\bin\client-agent-core.exe --notify
.\bin\client-agent-core.exe --run-once
.\bin\client-agent-core.exe --startup-status
.\bin\client-agent-core.exe --enable-startup
.\bin\client-agent-core.exe --disable-startup
.\bin\client-agent-core.exe --tray
.\bin\client-agent-core.exe --settings-window
.\bin\client-agent-core.exe --service-status
.\bin\client-agent-core.exe --service-install
.\bin\client-agent-core.exe --service-start
.\bin\client-agent-core.exe --service-stop
.\bin\client-agent-core.exe --service-uninstall
.\bin\client-agent-core.exe --update-check
.\bin\client-agent-core.exe --update-download
.\bin\client-agent-core.exe --update-apply
```

说明：
- 普通用户正式使用优先通过 `management-server.exe` 打开的 Web Admin 操作；上述参数保留给维护、自动化和烟测。
- `--setup` 打开本机配置文件。
- `--open-log` 打开 `logs/client-agent.log`。
- `--notify` 执行一次状态刷新后弹出通知气泡。
- `--run-once` 执行一次状态刷新并输出 JSON，供测试和维护使用。
- `--monitor` 会周期上报状态、轮询 Server 消息、写入本地日志并弹出通知。
- 开机启动使用当前用户注册表 `HKCU\Software\Microsoft\Windows\CurrentVersion\Run`，注册项名称为 `WoW Client Agent`，启动命令为当前 `client-agent.exe`。移动发布包目录后需要重新执行 `--enable-startup`。
- `--tray` 启动当前用户 Session 的托盘常驻 UI，右键菜单可控制 monitor、设置、日志、开机启动、Service 和更新。
- `--settings-window` 打开本机配置编辑窗口。
- `--service-install/start/stop/status/uninstall` 管理 Windows Service `WoWClientAgent`；安装和启停通常需要管理员权限。
- `--update-check/download` 检查或下载 GitHub 最新发布包。
- `--update-apply` 检查新版、下载发布包，并启动独立 updater；updater 会停止安装目录下旧进程、备份旧文件、保留 `data/logs/updates`，再覆盖新发布包并重启 Client。

## 本机控制中心
P22 起，发布包提供本机控制中心入口：

```powershell
.\WoW-Manager.exe
```

控制中心能力：
- 安装 / 修复。
- 启动 Server。
- 启动 Client。
- 打开 Web 管理页。
- 打开日志目录。
- 打开程序目录。
- 卸载程序。

安装 / 修复动作：
- 复制发布包到 `%LOCALAPPDATA%\WoWFramework`。
- 创建桌面快捷方式 `WoW Framework`、`WoW Server` 和 `WoW Client`。
- 创建开始菜单目录 `WoW Framework`。
- 快捷方式会使用 Server/Client 图标。

卸载入口：

```powershell
.\WoW-Remove.exe
```

卸载动作也可以从控制中心执行，会删除程序文件和快捷方式，并保留 `data`、`logs` 目录。

## 启动 Web Admin
```powershell
cd 实现模块/web-admin
npm run dev -- --port 5173
```

浏览器访问：

```text
http://127.0.0.1:5173
```

Web Admin 当前读取 Management Server 的健康检查和 Client 状态 API。生产前必须补鉴权、TLS 和 CORS 白名单。

## 一键本机运行
P10 起提供本机一键运行入口：

```powershell
.\tools\run-local.cmd
```

该入口会启动 Management Server，并启动 Client monitor。两个窗口保持运行，关闭窗口即可停止对应进程。

高级参数仍通过独立脚本控制：

```powershell
.\tools\start-server.ps1 -HostAddress 127.0.0.1 -Port 18080 -OpenBrowser
.\tools\start-client.ps1 -ServerHost 127.0.0.1 -ServerPort 18080 -Monitor
```

使用大漠时必须切换到 `x86` Client，并提供 32 位 `DmBridge.dll` 路径：

```powershell
.\tools\start-client.ps1 -ClientArch x86 -DmBridgePath .\dm-bridge\Win32\DmBridge.dll
```

## 编译 DmBridge
```powershell
.\实现模块\dm-bridge\build.ps1
```

输出文件：

```text
target/dm-bridge/Win32/DmBridge.dll
```

该输出目录不进入 GitHub。大漠二进制、CHM/CHW、授权资料和私有配置也不得入库。

## Server 消息
Web Admin 的 Client 详情面板可发送 Server 消息。Client monitor 收到后会：
- 写入 `logs/client-agent.log`。
- Windows 下弹出右下角通知气泡。

也可直接调用 API：

```powershell
Invoke-RestMethod -Method Post `
  -Uri 'http://127.0.0.1:18080/api/client/messages/local-dev-client' `
  -ContentType 'application/json' `
  -Body '{"title":"测试消息","body":"hello"}'
```

## Web Admin 远程操作
v1.11.0 起，Web Admin 左侧 `远程操作` 是日常管理 Client 的正式入口，可下发：
- `script.run_bootstrap`：让 Client 重新执行本机已配置并通过安全校验的 `scripts/bootstrap.lua`。
- `script.deploy_bundle`：把 Web Admin 输入的 Lua 内容热推送到勾选 Client 的 `scripts/` 目录。
- `script.start`：启用 Lua，并立即执行当前 bootstrap。
- `script.stop`：停用下一轮 Lua 执行，Client monitor 继续在线和接收命令。
- `script.status`：查询 Client 当前 Lua 开关、脚本路径、安全门和权限摘要。
- `update.apply`：让 Client 检查 GitHub Release、下载新版包并安排自替换。
- `update.check` / `update.download`：只检查或只下载更新包。
- `service.status/install/start/stop`：管理 `WoWClientAgent` Windows Service。
- `startup.status/enable/disable`：管理当前用户开机启动。
- `settings.open` / `log.open` / `tray.open`：在 Client 本机打开设置、日志或托盘。

命令流转方式：
- Web Admin 只把白名单命令写入 Server 队列。
- Client monitor 下次轮询后在本机执行命令。
- v1.17.0 起，Client monitor 执行后会向 Server 上报成功或失败回执。
- Web Admin 选中单台 Client 时，可在远程操作页查看最近执行回执。
- v1.23.0 起，配置 `MANAGEMENT_SERVER_AUDIT_PATH` 后，消息、命令和回执摘要会写入 Server 审计 JSONL。
- v1.25.0 起，Web Admin 会拦截 15 秒内同一批目标、同一命令、同一 payload 的重复下发。
- 生产审计仍需数据库、登录鉴权、角色权限和操作员身份。

## Client 使用 Lua 与 DM
Client 使用 Lua 的固定流程：
1. Client 启动或 monitor 刷新时读取 `config/client-agent.toml`。
2. `lua.bootstrap_path` 指向要执行的 Lua 文件，默认是 `scripts/bootstrap.lua`。
3. v1.25.0 内部测试包默认关闭 `script_security`，直接执行当前 Lua 文件；如果后台重新启用安全门，Client 会先校验 manifest、签名、SHA-256 和权限。
4. 校验通过后，脚本进入 `mlua` 宿主执行。
5. 执行结果会体现在 Client 状态上报和本机日志中。

DM 模式必须同时满足：
- 使用 x86 Client。
- 发布包内存在 `dm-bridge/Win32/DmBridge.dll`。
- v1.24.0 起，总包和 Client 分包内存在 `dm-bridge/Win32/dm.dll` 与 `dm-bridge/Win32/RegDll.dll`。
- 使用机器已完成大漠 COM 注册和授权。
- `config/client-agent.toml` 的 `dm.bridge_path` 指向该桥接 DLL。
- 需要 DM 的脚本在开启安全门时，manifest 的 `permissions` 必须包含 `dm.access`。
- `config/client-agent.toml` 的 `script_security.allowed_permissions` 也包含 `dm.access`。

P33 起 Web Admin 可在内部测试模式下直接把 Lua 内容热推送到勾选 Client。默认不要求 manifest；需要恢复校验时，在推送面板打开 manifest 校验并提供 manifest 内容。

## 发布包与分包
v1.12.0 起，打包脚本会生成三类 zip：

| 包 | 用途 |
|----|------|
| `WoW_Framework_vX.Y.Z_windows.zip` | 总包，包含 Server、Client、当前用户安装器、卸载器、工具脚本、公开文档和 Client DM 运行 DLL |
| `WoW_Server_vX.Y.Z_windows.zip` | Server 分包，只包含 Management Server、Server core 和 Server 工具脚本 |
| `WoW_Client_vX.Y.Z_windows.zip` | Client 分包，包含 Client Agent、Client core、DmBridge、DM 运行 DLL、配置、脚本和 Client 工具脚本 |

总包继续用于当前用户安装器和自动更新兼容。Client updater 仍下载 `WoW_Framework` 总包，不下载 Server/Client 分包。

Server 分包启动：

```powershell
.\management-server.exe
```

Server 分包查看控制台日志：

```powershell
.\bin\management-server-core.exe --no-open-browser
```

Client 分包启动：

```powershell
.\client-agent.exe
```

Client 分包维护入口：

```powershell
.\bin\client-agent-core.exe --run-once
.\bin\client-agent-core.exe --monitor
.\bin\client-agent-core.exe --service-status
.\bin\client-agent-core.exe --update-apply
```

发布包内 `config/client-agent.toml` 已将 `dm.bridge_path` 指向 `dm-bridge/Win32/DmBridge.dll`，从包根目录运行时不需要再手动改成开发目录路径。

发布包边界：
- 总包和 Client 分包包含 `dm-bridge/Win32/dm.dll`、`dm-bridge/Win32/RegDll.dll`。
- Server 分包不包含 Client、DmBridge、脚本和 DM 运行 DLL。
- 所有发布包都不包含授权文件、账号资料、签名私钥、真实商业脚本和 CHM/CHW 文档。

所有发布包都不包含：
- CHM/CHW 文档
- 授权文件
- 真实账号资料
- 签名私钥
- 真实商业脚本

## 脚本安全门
v1.25.0 内部测试包默认关闭 `script_security`，用于快速测试管理端热推送和 DM 脚本。后台重新启用后规则如下：
- Lua 文件必须匹配 manifest 中的 SHA-256。
- manifest 必须通过 Ed25519 签名验证。
- manifest 请求权限必须包含在本机允许权限中。
- 未授权 API 不注册到 Lua 全局环境。

如果 `script_security.enabled = true`，修改 `scripts/bootstrap.lua` 后必须同步更新 manifest hash 和签名，否则 Client Agent 会拒绝启动。这是预期行为。内部测试时保持 `script_security.enabled = false` 即可直接修改和热推送脚本。
