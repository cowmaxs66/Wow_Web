# DM 实机烟测与多机通讯规划

## 阶段定位
P28 用于把 DM 能力从“桥接层单元/COM 烟测”推进到“Client 可切换 Lua 脚本实测”。同时提前规划多机器管理和通讯效率，避免后续在线客户端数量增加后再返工。

## DM 烟测脚本
`dm_smoke.lua` 只执行以下动作：
1. 读取 DmBridge ABI 版本。
2. 初始化大漠 COM。
3. 读取大漠版本。
4. 读取屏幕 `(0, 0)` 颜色。
5. 读取最后大漠错误码。
6. 关闭 DmBridge。

脚本返回形如：

```text
dm_smoke|abi=1|ver=7.2149|color00=000000|last_error=0
```

## DM 烟测前置条件
- Client 必须使用 x86 core。
- `dm-bridge/Win32/DmBridge.dll` 必须存在。
- 本机大漠 COM 必须可创建 `dm.dmsoft`。
- 切换 DM smoke 时，`trusted_signer_public_key` 必须使用 `ea4a6c63e29c520abef5507b132ec5f9954776aebebe7b92421eea691446d22c`。
- 发布包不会包含 `dm.dll`、`RegDll.dll`、授权文件、账号资料和商业脚本。
- 如大漠未注册，烟测应失败并返回明确错误，而不是静默跳过。

## 多机器管理规划
| 方向 | 设计要求 |
|------|----------|
| Client ID | 每台机器必须唯一，建议格式 `区域-机器名-用途-序号` |
| 分组 | 后续为 ClientStatus 增加 group/tags，Web 支持按分组筛选和批量命令 |
| 标签 | 用于区分机房、账号池、脚本类型、DM 能力、版本 |
| 批量操作 | Server 侧应有明确目标集合，禁止默认“全体执行危险命令” |
| 在线判断 | 当前 30 秒 stale 适合小规模测试，多机时应跟随心跳间隔动态计算 |
| 审计 | 多机操作必须补操作者身份、命令来源、目标范围、执行回执持久化 |

## 通讯效率评估
当前 Client monitor 每轮可能产生 3 类 HTTP 请求：
- 状态上报：`POST /api/client/status`
- 消息拉取：`GET /api/client/messages/{client_id}`
- 命令拉取：`GET /api/client/commands/{client_id}`

默认间隔 10 秒时估算：

| Client 数 | 当前请求量估算 | 风险 |
|-----------|----------------|------|
| 10 | 约 3 req/s | 可接受 |
| 100 | 约 30 req/s | 可接受但有冗余 |
| 1000 | 约 300 req/s | 需要优化 |

## 后续优化路线
1. 给轮询增加 jitter，避免多台 Client 同秒集中请求。
2. 新增 `/api/client/sync`：一次请求完成状态上报、消息拉取、命令拉取和回执上报。
3. Web 列表 API 增加分页、筛选和摘要，避免一次返回全部历史。
4. 命令和回执从内存队列迁移到 SQLite/Postgres。
5. 大规模场景再考虑 WebSocket 或长轮询；不在当前小规模试运行阶段提前引入复杂连接管理。
