# 历史状态与趋势分析设计

## 目标
P8 建立最小历史状态闭环，让 Web Admin 能展示选中 Client 的最近上报历史和趋势概览。

## 设计边界
- 历史存储先放在 Management Server 内存中。
- 每个 Client 只保留最近 50 条状态。
- Server 重启后历史清空。
- 不引入数据库、迁移、索引、备份和生产持久化。
- 不伪造历史样本，Web 只展示 API 返回的真实上报记录。

## Server API
| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/client/history/{client_id}` | 返回指定 Client 的历史状态 |

## 响应结构
| 字段 | 说明 |
|------|------|
| `client_id` | 查询目标 Client |
| `limit` | 单 Client 历史保存上限 |
| `total` | 当前返回数量 |
| `items` | 按时间升序排列的状态信封 |

## Web 展示
| 区域 | 内容 |
|------|------|
| 历史趋势 | 最近上报次数、首末时间、在线样本数、最近脚本 |
| 时间线 | 按时间列出真实状态样本 |
| 迷你趋势线 | 根据历史点位绘制简单 SVG 折线，不伪造缺失点 |

## 后续持久化条件

- 数据库选型。
- 状态表 schema。
- 索引策略。
- 数据保留策略。
- 清理任务。
- 备份与恢复。
