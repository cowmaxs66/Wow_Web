# 多机器管理与通讯效率设计

## 阶段范围

本设计覆盖 P29 与 P30：

- P29：多机器管理资料模型、Client 分组/标签、批量操作安全边界。
- P30：通讯效率优化，先增加 monitor jitter，再增加 `/api/client/sync` 合并同步接口。

## 设计结论

- `client.id` 继续作为稳定机器身份，不允许 Server 远程修改。
- `display_name`、`group`、`tags` 作为 Client 身份展示层字段，写入 `client-agent.toml` 并随状态上报。
- Web Admin 按分组和标签展示 Client，便于多机器扫描，不引入复杂数据库筛选。
- 批量下发需要显式确认；单台操作不增加阻力。
- monitor 周期增加 jitter，降低多机器同时启动时的请求尖峰。
- `/api/client/sync` 合并状态上报、消息拉取和命令拉取，减少每轮 HTTP 往返。

## Client 身份模型

```toml
[client]
id = "local-dev-client"
display_name = "Local Dev Client"
group = "default"
tags = ["local", "test"]
```

字段说明：

| 字段 | 说明 | 远程可改 | 备注 |
|------|------|----------|------|
| `id` | 稳定 Client ID | 否 | 用于历史、命令、回执关联 |
| `display_name` | UI 显示名称 | 是 | 不能为空 |
| `group` | 分组 | 是 | 不能为空，默认 `default` |
| `tags` | 标签列表 | 是 | 去空、去重后展示 |

## 批量操作安全边界

批量操作指一次选择多台 Client 下发消息或命令。

当前安全边界：

- Web 选择“全部客户端”时必须勾选确认。
- 命令仍只允许共享白名单中的固定命令。
- `config.apply` 仍只能修改白名单字段，不允许改 `client.id`。
- Server 写入队列不等于执行完成，结果以 Client 命令回执为准。

暂不纳入本阶段：

- 操作员登录、角色权限和审批流。
- 长期审计数据库。
- 按标签自动批量命令规则。

## 通讯效率

旧 monitor 每轮最多请求：

1. `POST /api/client/status`
2. `GET /api/client/messages/{client_id}`
3. `GET /api/client/commands/{client_id}`

新 monitor 优先请求：

```http
POST /api/client/sync
```

请求体：

```json
{
  "status": {
    "schema_version": 1,
    "message_type": "status",
    "client_id": "local-dev-client",
    "data": {}
  }
}
```

响应体：

```json
{
  "ack": {},
  "messages": {},
  "commands": {}
}
```

边界：

- `messages` 仍保留列表语义，不在本阶段做送达确认。
- `commands` 仍保留取出即清空语义，避免重复执行。
- sync 失败时 Client 会回退旧轮询链路，降低新旧版本混用风险。
