# 工程化地基修补设计

## 阶段定位
P25 面向工程可维护性，不新增用户功能。目标是把 P0-P24 累积出的主要工程风险先收敛：
- 远程命令白名单只有一份来源。
- `management-server/src/app.rs` 不再同时承载路由、校验和大体量测试。
- Client 远程命令错误有明确类型。
- GitHub Actions 能在远端自动执行基础验证。

## 改动边界
| 项目 | 决策 |
|------|------|
| 目标版本 | v1.18.0 |
| 协议 JSON | 不改变字段形状，继续使用 `command_type: String` |
| 命令清单 | 在 `shared-types` 定义常量和 `REMOTE_COMMAND_TYPES` |
| Server 校验 | 通过 `is_supported_remote_command` 判断命令类型 |
| Client 执行 | 通过共享常量匹配命令分支 |
| 错误类型 | `RemoteCommandError` 区分不支持命令和执行失败 |
| CI | 远端跑 Rust 与 Web 基础验证 |

## 模块拆分
| 文件 | 职责 |
|------|------|
| `management-server/src/app.rs` | 路由、处理器和上线日志 |
| `management-server/src/app_validation.rs` | API 请求校验 |
| `management-server/src/app_tests.rs` | app 路由与状态行为测试 |

## CI 范围
GitHub Actions 运行：
- `cargo fmt --all --check`
- `cargo clippy --workspace -- -D warnings`
- `cargo test --workspace`
- `npm ci`
- `npm run build`

CI 不运行：
- DmBridge Delphi 编译。
- `tools/package-release.ps1`。
- GitHub Release 资产上传。

原因：DmBridge 依赖本机 RAD Studio 与私有大漠环境，不能假设 GitHub runner 具备。

## 后续边界
P25 只完成“命令清单单一来源”的过渡方案。真正的命令 registry、trait 化命令、依赖注入、前端 Router/Pinia/Vitest/ESLint，后续阶段再拆。
