# 总体方案

## 项目目标
建立一套可复用的 Windows 自动化代理框架，支持：
- Rust 客户端稳定执行。
- Lua 作为业务脚本语言。
- 大漠插件通过桥接层接入。
- 后续通过 Web 管理端集中配置、监控和控制。

## MVP 收敛原则
第一阶段不直接做完整平台，先验证最小技术闭环：
- Rust workspace 可编译。
- Client Agent 可生成标准状态消息。
- Client 和 Server 共享协议类型。
- 文档、进度、风险、变更记录完整。

## 阶段划分
| 阶段 | 名称 | 目标版本 | 说明 |
|------|------|----------|------|
| P0 | 项目初始化与规范建立 | v0.1.0 | 建立目录、文档、版本管理、最小 Rust workspace |
| P1 | Client Agent 最小闭环 | v0.2.0 | 本地 Lua 宿主、状态上报结构、基础脚本执行 |
| P2 | 大漠桥接验证 | v0.3.0 | Delphi DmBridge 与 Rust libloading 调用验证 |
| P3 | Server 通讯闭环 | v0.4.0 | Axum HTTP 状态上报、健康检查和状态查询 |
| P4 | Web 管理端 MVP | v0.5.0 | 客户端列表、健康状态、状态详情和连接设置 |
| P5 | 脚本管理与安全机制 | v0.8.0 | 脚本包、版本、签名、权限白名单 |
| P6 | 发布归档与交接 | v1.0.0 | 打包、部署、交付、归档 |

## 当前决策
- P0 只放最小可编译 Rust 骨架，不写大规模业务代码。
- DmBridge 和 Web Admin 先建立说明目录，避免占位代码污染。
- GitHub 远端默认使用 private 仓库；远端未确认前先做本地 git。
- v1.0.0 采用源码与文档归档，不发布包含大漠二进制或授权资料的安装包。
